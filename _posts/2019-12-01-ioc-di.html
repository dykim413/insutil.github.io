IoC 컨테이너 & DI Iversion of Control 은 Dependency Injection 이라고도 하며, spring application 에서는 객체의 생성, 관계설정, 사용, 제거 등의 작업을 코드가 아닌 독립된 컨테이너가 담당함을 뜻합니다. 그래서 스프링 컨테이너를 IoC 컨테이너 라고도 합니다. 스프링에선 IoC를 담당하는 컨테이너를 빈 팩토리 또는 애플리케이션 컨텍스트 라고 칭합니다. 스프링 컨테이너 = IoC 컨테이너 = 빈 팩토리 = 애플리케이션 컨텍스트 라고 봐도 크게 틀리지 않지만 객체의 생성과 관계 설정 이라는 DI 관점에서 볼때는 스프링 컨테이너를 빈 팩토리 라고 할 수 있고, 빈의 생성부터, 관계설정, 제거에 이르기까지 빈의 life cycle 관리 에 더해 기타 엔터프라이즈 애플리케이션 개발에 필요한 기능을 추가한 것을 애플리케이션 컨텍스트 라고 보시면 됩니다. BeanFactory ● 애플리케이션 컴포넌트를 관리하는 저장소 ● 빈 관련 메타설정 정보를 읽어들여 빈을 구성하고 제공 ● life cycle interface 지원 빈 팩토리에서 일어나는 빈의 생성, 초기화 인터페이스 와 순서는 아래의 그림을 참고 하면 됩니다. 위 그림에서 특히 관심이 가는것은 BeanPostProcesser 라는 빈 후처리 인터페이스 입니다. 여기서 빈 후처리란 빈이 최초에 생성되고 나서 빈의 설정을 후처리 함으로써 빈의 생명주기, 빈팩토리의 생명주기에 관여할 수 있게 함을 뜻합니다. 흔히 코딩을 할때 Autowired 어노테이션을 이용해서 빈을 DI 받습니다. 그런데 어떻게 어노테이션 하나만으로 빈이 적절한 시점에 DI 할 후보 빈을 선정하고 실제 빈을 주입하는지에 대해 궁금증을 가지신 분들이 계실지 모르겠는데.. 바로 빈 후처리기, BeanPostProcesser 에 의 후처리 될때 빈이 주입됩니다. 후처리 순서는 그림에서 보듯이 아래와 같습니다. ① BeanPostProcesser 의 postProcessBeforeInitialization (빈 초기화 전) ① InitializingBean's afterPropertiesSet ① custom init-method ① postProcessAfterInitialization ( 빈 초기화 후 ) ApplicationContext ● 스프링의 IoC 컨테이너는 일반적으로 애플리케이션 컨텍스트를 칭한다. ● BeanFactory 를 확장 (자체로 빈 팩토리 기능을 한다.) ● 메시지 소스 처리 기능, 이벤트 퍼블리싱, 리소스 로딩 등의 많은 추가 기능을 가진다. ApplicationContext 가 추가적으로 가지는 기능은 스프링 레퍼런스를 보면 알 수있습니다. 보다시피 BeanFactory를 구현하고 있고, 위에서 언급한 이벤트, 메시지, 리소스로더 관련 기능등을 구현하고 있습니다. (앞으로의 포스팅을 통해서 하나하나 전부 다뤄볼 예정입니다.) 지금까지 IoC 컨테이너 에 대해 최대한 간략하게 정리한 내용입니다. IoC 컨테이너의 가장 기초적이고 중요한 역할은 오브젝트를 생성하고 관리하는 것입니다. 즉 빈 을 관리하는것이 IoC 컨테이너의 가장 중요한 임무인 것이죠. IoC 컨테이너가 빈을 생성하고 관리하기 위해서는 POJO 클래스와 빈설정 메타정보가 필요합니다. 여기서 빈 설정 메타정보 구성은 전통적? 인 xml 파일을 이용하는 방법이 있고, 부트 기반으로 대부분의 자바 웹 애플리케이션을 제작하는 요즘은 어노테이션 기반으로 빈 설정을 합니다. 두방법 모두를 간단히 테스트 해보겠습니다. 단...... 다음 포스팅에서요~~ 오늘은 이만..
