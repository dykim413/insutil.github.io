IoC 컨테이너 & DI Iversion of Control 은 Dependency Injection 이라고도 하며, spring application 에서는 객체의 생성, 관계설정, 사용, 제거 등의 작업을 코드가 아닌 독립된 컨테이너가 담당함을 뜻합니다. 그래서 스프링 컨테이너를 IoC 컨테이너 라고도 합니다. 스프링에선 IoC를 담당하는 컨테이너를 빈 팩토리 또는 애플리케이션 컨텍스트 라고 칭합니다. 스프링 컨테이너 = IoC 컨테이너 = 빈 팩토리 = 애플리케이션 컨텍스트 라고 봐도 크게 틀리지 않지만 객체의 생성과 관계 설정 이라는 DI 관점에서 볼때는 스프링 컨테이너를 빈 팩토리 라고 할 수 있고, 빈의 생성부터, 관계설정, 제거에 이르기까지 빈의 life cycle 관리 에 더해 기타 엔터프라이즈 애플리케이션 개발에 필요한 기능을 추가한 것을 애플리케이션 컨텍스트 라고 보시면 됩니다. BeanFactory ● 애플리케이션 컴포넌트를 관리하는 저장소 ● 빈 관련 메타설정 정보를 읽어들여 빈을 구성하고 제공 ● life cycle interface 지원 빈 팩토리에서 일어나는 빈의 생성, 초기화 인터페이스 와 순서는 아래의 그림을 참고 하면 됩니다. ApplicationContext ● 스프링의 IoC 컨테이너는 일반적으로 애플리케이션 컨텍스트를 칭한다. ● BeanFactory 를 확장 (자체로 빈 팩토리 기능을 한다.) ● 메시지 소스 처리 기능, 이벤트 퍼블리싱, 리소스 로딩 등의 많은 추가 기능을 가진다. ApplicationContext 가 추가적으로 가지는 기능은 스프링 레퍼런스를 보면 알 수있다. 보다시피 BeanFactory를 구현하고 있고, 위에서 언급한 이벤트, 메시지 관련 기능등을 구현하고 있다. 지금까지 IoC 컨테이너 에 대해 최대한 간략하게 정리한 내용입니다. 이제부터는 실제로 IoC 컨테이너를 만들 빈을 설정하는 실습을 간단히 해보겠습니다. 사실 요즘은 부트를 쓰면서 거의 모든 설정이 어노테이션 기반으로 자동구성 되기 때문에 xml 을 사용해서 한땀한땀 빈을 설정할 일은 없지만 간단하게 전통적인? 방법으로 실습해보면서 감을 익혀보자.
